# 02 · 系统架构

> 一条信条贯穿全篇：**结构化内容与渲染分离**。课件是一份 JSON 数据模型，
> 渲染层确定性地呈现它。这条原则一肩三任——多端自适应、换模板而内容不变、原生互动。

## 分层架构

```
┌─────────────────────────────────────────────────────────────┐
│  表现层 Presentation（手机优先、多端自适应）                    │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────────────┐ │
│  │ 创作入口  │ │ 编辑器    │ │ 渲染/播放 │ │ (P2) 学生端       │ │
│  │ 一句话/导入│ │ 大纲·精修 │ │ Renderer  │ │  扫码加入/作答    │ │
│  └──────────┘ └──────────┘ └──────────┘ └──────────────────┘ │
├─────────────────────────────────────────────────────────────┤
│  应用层 Application（Next.js Route Handlers / Server Actions） │
│  生成编排 · 课件 CRUD · 模板服务 · PPT 导入 · 导出 · 鉴权       │
├─────────────────────────────────────────────────────────────┤
│  AI 流水线 Pipeline（多 Agent + 模型路由）                     │
│  意图解析 → 教学设计(大纲) → 内容生成 → 版式设计 → 校验          │
├─────────────────────────────────────────────────────────────┤
│  领域核心 Domain                                              │
│  课件 Schema（节/页/块/互动块）· 校验 · 模板/主题 · 渲染规则     │
├─────────────────────────────────────────────────────────────┤
│  基础设施 Infrastructure                                       │
│  PostgreSQL(JSONB) · 对象存储(S3) · LLM Provider · (P2)实时层   │
└─────────────────────────────────────────────────────────────┘
```

## 三条核心数据通路

### 1. 一句话 → 课件（生成闭环）

```
用户一句话
   │  POST /api/generate/intent
   ▼
[意图解析 Agent] ──► 要素卡片（学科/学段/主题/时长 + "我替你补的"）  ← 用户一眼纠偏 (F2)
   │  POST /api/generate/outline
   ▼
[教学设计 Agent] ──► 轻量大纲（节次 + 每节要点）  ← 用户增删改调序确认 (F3)
   │  POST /api/generate/content（渐进式流式返回）
   ▼
[内容生成 Agent] ──► 各节内容块
[版式设计 Agent] ──► 在所选模板下布局/配图
[校验 Agent]    ──► 事实/逻辑/学段自检
   ▼
完整课件 JSON ──► 存 Postgres(JSONB) ──► 渲染层呈现 (F4)
   │  POST /api/refine（块级局部修改）
   ▼
[精修] 自然语言指令 / 直接编辑 ──► 仅重写受影响的块，不洗牌全篇 (F8)
```

关键：**大纲确认是意图对齐的最高杠杆点**（秒级、便宜），全文生成只在大纲确认后才投入。

### 2. PPT → 课件（存量迁移，F6）

```
.pptx 上传 ──► [OOXML 解析] 抽取 文本/图片/结构
          ──► [映射器] 映射到课件 Schema（slide→page, shape→block）
          ──► 套用所选模板重排 ──► 课件 JSON（与通路 1 同构）
          ──► (进阶) LLM 深度改造：重组/补全/配图/建议互动
```

迁移产物与「一句话生成」产物 **完全同构**——都是同一份课件 Schema，因此后续编辑、播放、导出、互动全部复用。这是「结构化模型」作为统一枢纽的价值。

### 3. 课件 → 课堂（呈现与互动）

```
课件链接 /deck/[id]/play ──► 加载 JSON ──► 响应式渲染（各端重排） (F9)
                                              │
                          (Phase 2) 一键发起课堂 ──► WebSocket 会话
                                              ▼
                          学生扫码加入 ──► 推送投票/题目 ──► 作答聚合 ──► 大屏实时呈现
                                              ▼
                          学情数据回流 ──► Postgres
```

## 模块边界（MVP）

| 模块 | 职责 | 关键产物 |
| --- | --- | --- |
| `schema` | 课件数据模型与校验（领域核心，零依赖） | TS 类型 + Zod schema |
| `ai` | LLM 抽象、Agent 流水线、Prompt | Provider 接口、各 Agent |
| `renderer` | 把课件 JSON 确定性渲染为 React | 块渲染器、布局引擎 |
| `templates` | 模板/主题定义与切换 | 主题 token、模板注册表 |
| `editor` | 大纲编辑、块级精修、直接编辑 | 编辑器状态机（Zustand） |
| `player` | 全屏播放、翻页、演讲者备注/计时 | 播放控制 |
| `import` | .pptx 解析与映射 | OOXML 抽取、映射器 |
| `export` | PDF 导出 | Puppeteer 渲染管线 |
| `persistence` | 课件/用户存取 | node-postgres、migrations、Repository（src/lib/*-store.ts） |
| `auth` | 注册登录、作品空间 | Auth.js 配置 |

## 渲染层：自适应而非固定尺寸

课件 **不是** 16:9 的固定画布。每一页是一组内容块 + 一个布局意图（`layout`），渲染层按设备视口把块重新排布：

- **手机**：单列纵向流，块按序堆叠，字号/留白为小屏优化；
- **平板/桌面/大屏**：按 `layout` 呈现多栏、并排媒体等。

这让「同一份课件各端各得其所」成为渲染层的确定性行为，而非为每端各做一套。详见 [03-data-model.md](03-data-model.md)。

## 为互动预留（MVP 不做实时，但模型就绪）

互动块（poll/mcq/quiz/discussionWall…）从第一天起就是课件 Schema 的一等公民：MVP 可在课件里 **编排** 一道题并在播放时 **静态呈现**；Phase 2 接入实时层后，同一个互动块获得「发起→作答→聚合」的运行时能力，**课件结构无需改动**。这正是 PRD 强调的「架构从第一天起为互动就绪、日后无需返工」。

## 非功能性约束落点

| 需求 | 架构落点 |
| --- | --- |
| 大纲秒级、全文 1–2 分钟渐进式 | 模型分级路由；内容生成流式返回，逐节渲染 |
| 离线打开已生成课件 | Serwist 缓存课件 JSON + 静态资源 |
| 多端自适应 | 渲染层按视口重排 |
| 数据合规（PIPL） | 默认国产模型、境内存储；数据分区弹性 |
| 内容安全 | 校验 Agent + 人审兜底 + 准确性提示 |
